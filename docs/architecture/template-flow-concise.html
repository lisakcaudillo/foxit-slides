Slide platform · template lifecycle

Template flow: Figma to PowerPoint

The goal: turn a prompt and your materials into a beautiful, presentation-ready PowerPoint deck that stays fully editable. This page walks through how it works.

What makes it work

Design once

A layout is drawn and role-tagged in Figma once; every deck reuses it.

Fits the content

The generator picks the layout + combo that matches the content, never forced.

Grounded to your source

You set how closely it sticks to the source. The default uses only what you provide.

Judged before it ships

An objective gate plus two AI judges (words and picture), with a capped fix loop.

Native PowerPoint

Every element a real, editable object via Foxit Slidewright, matching the preview.

On-brand

The chosen theme's fonts and colors are applied consistently across the deck.

7 SELECT

Layout by content

content shape → best-fit combo

The content is decided first, then a layout is chosen to fit it. The number of items picks the combo, never forced.

How?
First the system decides what the slide will say: how many points, whether there is a number to feature, a table, or an image. Then it picks the layout built for that kind of content and that many items. Three supporting points fit a three-card layout; one big number fits a single-stat layout. A chart or table layout is never chosen when there is no real data to fill it.

Each slot gets text sized to fit its space. If it runs long it shrinks a step or tightens the line spacing; if it still doesn't fit, the AI rewrites it shorter as a full sentence, never cut off mid-word. The slot's padding is respected, so text never touches the edges.

The unhappy path: when content does not fit

The main safeguard is order: the content is decided first, then a layout is chosen to fit exactly that much, so slots are rarely too full or too empty. When something still does not fit, whether it overruns its box or leaves the slide sparse, each case has a defined fallback.

Text is longer than its boxIt is rewritten shorter as a full sentence, and the font shrinks a step or the line spacing tightens, until it fits. It is never clipped.
A card still overflows after thatThe objective check catches the overflow and the slide loops back for another fix, up to a set limit.
The text is far too long for the chosen layoutThe slide is moved to a roomier layout: a bigger box, or fewer items per row.
There are more or fewer items than the layout holdsA different combo or layout is picked to match the number of items.
A slot would be left emptyThe slot is dropped, or the slide moves to a smaller layout. Empty slots are never padded with filler.
A number or fact is not in the sourceThe slot is left blank rather than inventing one.
Two elements overlap or run off the slideThe check flags it and the element is nudged inward or shrunk to fit.

One design → one layout, saved with exact rules

The designer names each layer role/group. The role says what the slot holds (title, body, metric, table, image); the group ties repeating slots together. Roles are a fixed vocabulary, so the same tag means the same thing everywhere, and maps to a PowerPoint element on export.

This specific design is one layout. Its slots' exact geometry is captured and saved to the combo layout library, which generation reads. The same layout is stored as several combos: set combinations of blocks for different amounts of content (a cards layout at 2, 3, or 4 cards; body + image at three densities), and the generator picks the combo that fits each slide.

Every slot carries the hard rules generation must respect
  • Role: what it holds, and the PowerPoint element it becomes on export
  • Position & size: each slot's x, y and its maximum width and height, on the 960×540 frame
  • Padding: the outer safe margin plus card padding; text and content never cross it
  • Text budget: how much text fits, set per density (concise · detailed · extensive)

Here is that layout with the rules drawn on it: the safe margin, each slot's role, position (x, y) and size (width × height), the padding inside a card, the gaps between sections, and the character budget each text slot is written to.

Shaded band = the safe margin, where content is blocked. Dashed inset = card padding. Pills = the character budget for a text slot; text is written up to the budget and no further, then fit to the box. Widths and heights are box maximums; content is written to fit inside them. Glossary

RoleWhat a slot holds: title, body, metric, table, image. A fixed vocabulary.
GroupTies repeating slots together (e.g. the cards in a row).
SlotOne placed element on a layout, with a fixed position and size.
LayoutOne design: a fixed arrangement of slots on the 16:9 frame.
TemplateThe saved layout the generator fills: role-tagged slots with fixed geometry. Its combos are the variants.
ComboShort for combination: a set arrangement of content blocks (title + 3 cards, or body + image), saved at a given content count and density. Each layout has several; the generator picks the one that fits.
DensityHow full a slide is: concise · detailed · extensive. Sets text amount and which combo is used.
Text budgetHow much text a slot holds, set per density.
PaddingThe safe margins (around the frame and inside cards) that content never crosses.
ThemeThe overall look you pick for a deck (Mono Light, Volt, …): its fonts, colors, and decorations.
SkinHow a theme paints a layout: its fonts and colors applied to the slots. Theme = the look you pick; skin = how it's applied.
BackgroundThe backdrop behind the content: a color, gradient, or image, set by the theme.
GroundingA setting for how closely the deck sticks to your source. The default, source-only, uses nothing beyond what you provide; looser settings let the model add general context.
JudgesThe reviewers: a deterministic gate, a content judge (words), and a visual judge (picture).
