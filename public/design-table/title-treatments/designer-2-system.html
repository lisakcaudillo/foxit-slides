Title Treatment System

A cover should never look like one template recolored 42 ways. Instead of authoring 42 bespoke title pages, I treat the cover as a small deterministic system: six named title styles, and a single selection rule that reads each theme's own identity (category, light/dark, mood) to auto-assign one. Because the rule keys on three independent signals, browsing a row of themes produces natural variety — adjacent themes rarely land on the same style by accident. The editorial "hardcover" anchor is one of the six. Every auto-pick is overridable: right-click a slide → Try a different title. The logic reuses the shape of the existing deriveCoverTier() in cover-tiers.ts, extended from 3 tiers to 6 named styles.

The six title styles

Editorial CoverThe anchor. Eyebrow + hairline rule + large headline with one italic accent word + byline strip. Quiet, bookish authority.
MastheadHeavy top/bottom rules frame a tracked-out title like a newspaper banner. Structured, institutional.
StatementOne oversized title fills the frame, baseline-anchored. Maximum confidence, no chrome.
Chapter DividerCentered title with a number/kicker and symmetric rules above & below. Calm, sectioned.
Split PanelType on one half, the theme's motif on the other. Keeps words crisp where data themes need it.
SpotlightTitle floats over a full-bleed motif glow with a soft scrim. For dramatic dark / branded themes.

How a theme gets its title style

The selector reads three signals already on every Theme object — category, tone, and archetype — and returns one of the six styles. Three signals × six outcomes spreads themes across the family, so a browse feels varied. A tiny named-override map handles the few themes whose right style isn't obvious from signals alone (same pattern as THEME_COVER_TIER today).

Selection rule — titleStyleForTheme(theme), evaluated top-to-bottom (first match wins):
  1. Named override? A short per-theme map wins outright (e.g. aurora → Statement). Escape hatch for editorial judgment.
  2. archetype = editorial → if legal theme → Editorial Cover (the bookish anchor); else → Masthead (editorial business reads as a banner).
  3. archetype = cinematic → Spotlight (dark/branded drama → full-bleed motif glow).
  4. archetype = warm → Chapter Divider (centered, breathing, friendly).
  5. archetype = product → business → Split Panel (data themes keep type crisp); else → Statement (bold creative product covers).
  6. Fallback → Editorial Cover — never breaks, never an empty image box.
Because archetype drives the primary branch and category/tone refine it, the four archetypes already guarantee at least four distinct styles before overrides — and overrides push the look-alikes apart.

Override the auto-pick — "Try a different title"

Design intelligence picks a sensible default, but the user always decides. Right-click any cover thumbnail to open the same context menu that already hosts Try a different cover and Try a different layout — now with a Try a different title submenu listing all six styles. The current pick is checked; choosing another re-renders the cover in place, keeping the theme's colors, fonts, and motif. Plain-language labels throughout.

Why this interaction shape

  • Mirrors what exists. The same right-click menu already carries Try a different cover and Try a different layout — users learn one gesture, not three.
  • Suggested, not forced. The auto-pick is shown checked; "Reset to suggested" lets a user return to the system's choice after experimenting (satisfies overridable design intelligence).
  • Cheap to preview. Switching styles is a pure re-layout of the same theme tokens — no regeneration, no network call, instant on click.
  • Plain language. The menu says "Title style", not titleStyle / "tier" / "treatment archetype". Internal names never reach the surface (the OCR → "Grab Text" precedent).
  • Consistent with P-UX2. The picker only appears on intent (right-click) — covers stay calm by default.
